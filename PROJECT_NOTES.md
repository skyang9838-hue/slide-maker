# Slide Maker 프로젝트 정리

## 1. 만들고 싶은 것

사용자가 준비한 슬라이드별 개요나 원고를 바탕으로, 같은 스타일과 이야기 흐름을 가진 슬라이드 이미지 20~30장을 생성하는 Codex 워크플로를 만든다.

- PowerPoint 파일을 만드는 것이 목적이 아니다.
- 편집 가능한 PPT 요소보다, 글자가 포함된 완성형 슬라이드 이미지가 결과물이다.
- 이미지 한 장씩 요청하고 기다리는 불편을 줄이고, 여러 장의 제작을 한 번의 작업으로 맡기는 것이 핵심이다.
- OpenAI 이미지 생성 API를 직접 사용하는 독립 서비스가 아니라, 사용자가 로그인한 Codex의 이미지 생성 기능을 활용한다.

## 2. 배포 방식

프로젝트를 GitHub 저장소로 배포한다.

다른 사용자는 저장소를 내려받고, 자기 컴퓨터에서 해당 폴더를 Codex로 연 뒤 사용할 수 있어야 한다. 워크플로는 특정 사용자나 특정 OpenAI 계정에 묶이지 않고 저장소 안의 지침, 양식, 예시 자료로 전달한다.

각 사용자는 다음 조건을 충족해야 한다.

- 자신의 계정으로 Codex에 로그인되어 있어야 한다.
- 해당 Codex 환경에서 이미지 생성 기능을 사용할 수 있어야 한다.
- 저장소 폴더에 대한 접근을 허용해야 한다.
- 이미지 생성 한도와 동시 처리 수는 계정 및 환경에 따라 달라질 수 있다.

따라서 모든 환경에서 30장을 완전히 동시에 생성한다고 보장하지는 않는다. 대신 작업 상태를 저장하고, 중단되거나 실패한 슬라이드만 이어서 생성할 수 있는 구조를 목표로 한다.

## 3. 사용자 흐름

### 단계 1: 웹 챗봇에서 내용 기획

사용자는 웹 ChatGPT, Gemini 등 원하는 챗봇과 자유롭게 대화하며 발표 내용과 슬라이드 구성을 만든다.

- 처음부터 복잡한 전용 프롬프트를 사용하지 않는다.
- 사용자는 평소처럼 주제, 대상, 목적, 넣고 싶은 내용 등을 챗봇과 편하게 의논한다.
- 특정 챗봇에 종속되지 않는다.

### 단계 2: 정해진 형식으로 정리

기획이 끝나면 사용자는 저장소에 포함된 `SLIDE_FORMAT.md`를 웹 챗봇에 첨부하거나 그 내용을 복사해 붙여넣는다.

웹 챗봇에는 다음과 같은 취지로 요청한다.

> 지금까지 나눈 내용을 이 형식에 맞춰 최종 정리해줘. 이 결과를 이미지 생성용 Codex에 전달할 거야.

`SLIDE_FORMAT.md`는 대화 전체를 통제하는 거창한 프롬프트가 아니다. 이미 나눈 대화를 Codex가 안정적으로 읽을 수 있는 Markdown 형식으로 정리하는 짧은 출력 양식이다.

웹 챗봇은 사용자의 로컬 파일을 파일 이름만으로 읽을 수 없으므로, 사용자가 반드시 파일을 첨부하거나 내용을 직접 붙여넣어야 한다.

### 단계 3: Codex에 전달

사용자는 웹 챗봇이 출력한 최종 Markdown 전체를 복사해 이 저장소를 연 Codex 채팅에 붙여넣는다.

예상 요청 문구:

> 아래 슬라이드 개요를 바탕으로 이 저장소의 제작 지침에 따라 전체 슬라이드 이미지를 생성해줘.

### 단계 4: 개요 보존 후 이미지 생성

Codex는 전달받은 개요를 `.slide-maker/current-deck.md`에 저장하고 작업의 기준으로 사용한다. 가장 최근 결과 이미지는 사용자가 바로 가져갈 수 있도록 `결과물/` 바로 아래에 둔다. 새로운 전체 생성 전에 기존 이미지는 `지난결과/YYYY-MM-DD_HHMM-short-title/`로 자동 보관한다.

실행 폴더마다 `deck.md`를 남기는 이유:

- 생성 도중 중단되어도 이어서 작업할 수 있다.
- 특정 슬라이드만 수정하거나 재생성할 수 있다.
- 다른 컴퓨터에서도 같은 입력을 재현할 수 있다.
- Git으로 개요 변경 내용을 확인할 수 있다.
- 긴 채팅 내용에 의존하지 않고 작업 기준을 유지할 수 있다.

사용자가 직접 개요 파일이나 내부 기록을 만들고 편집하도록 요구하지 않는다. 사용자는 개요를 채팅에 붙여넣고, 참고 자료가 있으면 `넣을곳/`에 넣기만 한다.

## 4. 내용과 문구에 관한 원칙

웹 챗봇은 각 슬라이드의 제목과 화면에 들어갈 짧은 핵심 구절 2~4개를 만든다.

Codex는 슬라이드 이미지에 적합하도록 다음을 조정할 수 있다.

- 핵심 구절의 길이와 표현 다듬기
- 제목, 핵심 구절과 선택적 핵심 문구의 위계 설정
- 줄바꿈과 정보 배치
- 슬라이드 간 표현의 일관성

핵심 구절은 단어 하나의 나열이나 긴 설명문이 아니라 의미가 분명한 짧은 구절로 작성한다. 레이아웃을 이유로 2개 미만으로 임의 삭제하지 않는다. 원래 의미와 사실은 바꾸면 안 되며 숫자, 날짜, 인명, 고유명사, 인용문 등 정확성이 필요한 정보는 임의로 변경하지 않는다.

## 5. 디자인에 관한 현재 결정

기본 시각 표현은 현대적인 편집 일러스트로 한다. 사용자가 디자인 스타일을 글로 설명하거나 예시 슬라이드 이미지를 함께 제공하면 전체 일러스트의 색감, 질감과 구도에 반영할 수 있다.

예:

- 미니멀하고 현대적인 스타일
- 고등학생 대상의 밝고 친근한 분위기
- 잡지 편집 디자인처럼 강한 타이포그래피
- 차분하지만 딱딱하지 않은 색상

예시 슬라이드, 로고와 사진은 구분하지 않고 모두 `넣을곳/`에 넣는다. 예시 슬라이드는 스타일 참고 자료로 사용한다. 실제 사진은 개요에서 `provided_photo`로 지정하거나 사용자가 명시적으로 요청한 경우에만 결과에 사용한다. 사진이 없으면 사진풍 현장 이미지를 새로 만들지 않고 일러스트나 다이어그램으로 전환한다. 참조 이미지 속 텍스트는 내용의 기준으로 삼지 않으며 개요의 필수 문구와 사실 정보를 우선한다.

## 6. 저장소가 담당할 역할

최종 저장소에는 다음 성격의 자료가 포함될 예정이다.

- Codex가 자동으로 읽는 저장소 작업 지침 `AGENTS.md`
- 반복 가능한 슬라이드 이미지 생성 워크플로
- 웹 챗봇 결과를 정리하기 위한 `SLIDE_FORMAT.md`
- Codex가 내부적으로 저장하는 최종 개요와 실행 기록
- 디자인 스타일 지침
- 사용자가 자료를 한곳에 넣는 `넣을곳/` 폴더
- 가장 최근 이미지를 바로 가져가는 `결과물/` 폴더
- 이전 명령의 이미지를 자동 보관하는 `지난결과/` 폴더
- 진행 상태 및 실패한 슬라이드를 기록하는 방법
- 처음 사용하는 사람을 위한 짧은 사용 설명

사용자용 폴더 구조는 `넣을곳/`, `결과물/`, `지난결과/`로 확정했다. 내부 개요와 실행 기록은 `.slide-maker/`에서 Codex가 관리한다.

## 7. 아직 결정하지 않은 내용

다음 대화에서 차례로 정할 사항:

- Codex가 생성 전에 전체 디자인 방향을 사용자에게 확인받을지
- 이미지 생성 작업을 몇 장씩 나눌지
- 이미지 안의 글자 오류를 검사하고 수정하는 절차
- 여러 슬라이드 사이의 시각적 일관성을 유지하는 방법
- 다른 사용자의 Codex 환경에서 이미지 생성 기능이 없을 때 안내하는 방법

## 8. 현재 한 문장 정의

> 웹 챗봇과 자유롭게 기획한 슬라이드 개요를 정해진 Markdown 형식으로 Codex에 전달하면, Codex가 이를 보존하고 같은 스타일의 완성형 슬라이드 이미지 20~30장을 생성하는 GitHub 배포형 워크플로.
