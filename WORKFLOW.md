# Codex Workflow

## 1. 입력 확인

1. 사용자가 채팅에 붙여넣은 최종 Markdown을 `.slide-maker/current-deck.md`에 저장합니다.
2. `넣을곳/`의 모든 파일을 확인합니다.
3. PPT 캡처, 예시 슬라이드와 무드보드는 스타일 참고 자료로 우선 판단합니다.
4. 로고, 제품 사진과 인물 사진은 결과물에 실제로 사용할 자료로 우선 판단합니다.
5. 용도가 불분명하고 잘못 판단하면 결과가 크게 달라지는 경우에만 사용자에게 확인합니다.

참고 이미지 속 문구보다 개요의 `must_keep`과 `required_text`를 우선합니다. 예시 디자인을 그대로 복제하지 않고 색감, 여백, 구도, 타이포그래피와 정보 밀도 원칙을 전체 덱에 일관되게 적용합니다.

## 2. 기존 결과 자동 보관

새로운 전체 생성을 시작하기 전에 `결과물/`을 확인합니다.

1. 슬라이드 이미지가 없으면 바로 새 생성을 시작합니다.
2. 이미지가 있으면 `.slide-maker/current-run.md`에 기록된 실행 이름을 확인합니다.
3. 실행 이름을 알 수 없으면 현재 시각과 `previous-result`를 사용합니다.
4. `지난결과/YYYY-MM-DD_HHMM-short-title/` 폴더를 만들고 기존 이미지를 그곳으로 옮깁니다.
5. 같은 폴더명이 있으면 `-02`, `-03`을 붙이며 기존 파일을 덮어쓰지 않습니다.

## 3. 새 실행 준비

1. 현재 시각과 발표 제목으로 `YYYY-MM-DD_HHMM-short-title` 실행 이름을 만듭니다.
2. `.slide-maker/runs/<실행 이름>/`에 `deck.md`, `progress.md`, `prompts/`를 준비합니다.
3. `.slide-maker/current-deck.md`를 실행 기록의 `deck.md`로 복사합니다.
4. `.slide-maker/current-run.md`에 현재 실행 이름을 기록합니다.
5. `progress.md`에 시작 시간, 참고 파일 목록과 슬라이드 상태를 기록합니다.

## 4. 이미지 생성

1. 이미지는 `결과물/slide-01.png`, `slide-02.png` 형식으로 바로 저장합니다.
2. 실제 사용한 프롬프트는 내부 실행 기록의 `prompts/slide-01.md` 형식으로 저장합니다.
3. 각 프롬프트에는 전체 스타일, 슬라이드 역할, 핵심 메시지와 필수 문구를 포함합니다.
4. 이미지 안의 글자는 짧고 명확하게 유지합니다.

## 5. 상태와 검수

각 슬라이드 상태는 내부 실행 기록의 `progress.md`에 `pending`, `generated`, `needs_fix`, `failed`, `skipped` 중 하나로 기록합니다.

생성 후 다음을 확인합니다.

- 필수 문구, 숫자, 날짜, 인명과 고유명사가 정확한지
- 슬라이드 간 시각 스타일이 일관적인지
- 텍스트가 읽기 어렵거나 지나치게 많지 않은지
- `결과물/`의 파일명이 순서대로 저장되었는지

## 6. 이어서 만들기와 수정

- 특정 결과의 수정, 재생성 또는 이어서 만들기는 새 전체 생성으로 취급하지 않습니다.
- 현재 결과라면 `결과물/`과 `.slide-maker/current-run.md`를 사용합니다.
- 지난 결과라면 사용자가 지정한 `지난결과/<실행 이름>/`을 사용합니다.
- `pending`, `failed`, `needs_fix` 상태와 실제 이미지 파일을 비교해 필요한 슬라이드만 처리합니다.
- 비교가 필요하면 `slide-01-v2.png`처럼 버전을 붙이고, 아니면 같은 파일명으로 교체합니다.
